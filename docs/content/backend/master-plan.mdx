# Backend Master Plan

## Goal

Make backend contracts and execution semantics explicitly support a training-first TFRecord schema where image payload defaults to a bytes feature.

## Architecture Objective

Backend should act as the policy and contract layer between frontend workflows and `hephaes` conversion internals. The backend must expose a stable, inspectable schema contract that clearly communicates image payload representation and guards users from silent behavior drift.

## Target Backend Responsibilities

1. Publish and version conversion-schema metadata, including image-field representation rules.
2. Validate conversion requests against schema policy before executing long-running jobs.
3. Persist conversion intent and effective schema in job and manifest metadata for auditability.
4. Expose compatibility information for older outputs where image payload may be list-based.
5. Provide actionable API errors and warnings when payload representation is ambiguous.

## API And Contract Direction

1. Add explicit output-schema fields in authoring and execution responses that state how image payload is encoded.
2. Include an effective-policy object in conversion jobs so consumers can detect defaulted behavior.
3. Add compatibility markers for legacy datasets to distinguish bytes-based vs sequence-based image fields.
4. Keep response shape stable and additive so frontend adoption is low risk.

## Persistence And Traceability

1. Store requested output settings and resolved schema policy per conversion run.
2. Record schema version and compatibility mode in manifest-linked backend metadata.
3. Ensure historical runs remain queryable by representation type for migration planning.

## Ownership Boundary

- `hephaes` owns conversion semantics and low-level payload encoding behavior.
- Backend owns externally visible contract shape, policy exposure, persistence, and migration signaling.
- Frontend owns presentation and guided user actions based on backend contracts.

## Success Criteria

1. Any API consumer can determine image payload representation without opening TFRecord data directly.
2. Conversions using default settings are unambiguous and training-oriented.
3. Legacy outputs are discoverable and clearly flagged for compatibility handling.
