# Frontend Current State

## Purpose

Use this file as the quick implementation map for the frontend. It describes what is actually wired today, where the active routes and data hooks live, and which code paths are still only placeholders or future-work scaffolding.

## Snapshot

The frontend is a Next.js App Router app built around the current backend APIs for dashboard, inventory, conversion execution, outputs, jobs, and replay.

It already supports:

- dashboard rollups and trend views
- inventory management with upload, directory scan, filters, bulk indexing, and tag editing
- asset detail pages with metadata, topics, episodes, jobs, conversions, convert links, and replay links
- a dedicated `/convert` authoring workspace for inspection, draft generation, preview, saved-config management, and backend-managed execution handoff
- jobs history and detail pages
- outputs catalog and output detail pages with metadata refresh actions
- replay timeline and sample browsing with websocket replay plus REST fallback

The conversion authoring workflow is now wired through the frontend. The remaining work is mostly visual polish and optional product decisions such as import/export affordances.

Recent contract integration updates:

- Frontend API types now model backend representation policy metadata on conversion, job, and authoring responses.
- Conversion request typing now includes explicit TFRecord image payload contract selection support (`bytes_v2` and `legacy_list_v1`).
- Frontend helper utilities now provide policy-version defaults, legacy-compatibility marker checks, and guarded output-contract normalization for mixed backend rollouts.
- Conversion authoring now shows explicit image payload contract callouts, training-first defaults, legacy compatibility warnings, and preview-time representation fields.
- Conversion status, job detail, outputs list, and output detail pages now surface effective image payload contract details for faster post-run verification.
- Outputs deep links now preserve schema context via query params (including image payload contract) across refresh/navigation.
- Authoring policy callouts now include explicit empty/error states for missing backend schema metadata to support mixed-version rollout resilience.
- Frontend now has Vitest coverage for representation-policy helpers, schema callout rendering, output preview payload facts, and deep-link schema context helpers.
- Frontend README now documents the user-facing image payload contract experience and screenshot checklist for release readiness.

## Stack And Global Structure

- `frontend/package.json`: Next.js 16, React 19, SWR, shadcn/Radix, Sonner, Tailwind CSS, and `@rerun-io/web-viewer`
- `frontend/app/layout.tsx`: root layout and global providers
- `frontend/components/app-shell.tsx`: top-level app shell and nav for Dashboard, Inventory, Outputs, and Jobs
- `frontend/lib/api.ts`: typed backend client contracts and request helpers
- `frontend/hooks/use-backend.ts`: central SWR hooks and cache invalidation helpers
- `frontend/components/ui/`: shared primitives

There is no dedicated frontend test suite in the repository right now.

### Inventory

- `frontend/app/inventory/page.tsx`
- `frontend/app/inventory/inventory-page.tsx`

- `frontend/app/inventory/inventory-upload-dialog.tsx`
- `frontend/app/inventory/inventory-scan-dialog.tsx`

Implemented behavior:

Implementation note:

- some filters are server-backed through `GET /assets`, while others remain client-side only
- there is no hard backend cap in the inventory listing route; visible rows can be lower than registered count when filters are active or when uploads are skipped (duplicate/invalid)

### Conversion execution

- `frontend/app/convert/page.tsx`
- `frontend/app/convert/new/page.tsx`
- `frontend/app/convert/use/page.tsx`
- `frontend/app/convert/conversion-authoring-workspace.tsx`
- `frontend/lib/conversion-authoring.ts`

Implemented behavior:

- root `/convert` now resolves to `/convert/use` when saved configs exist and `/convert/new` when they do not
- `/convert/new` is a sequential create wizard that starts with source selection, then reveals inspect, draft, spec editing, and preview one step at a time
- the create wizard uses a breadcrumb trail in the header area for step navigation, keeps source selection lean, avoids a selected-count chip in the title area, starts the inspect accordions collapsed, and places `Include preview` inline with the label feature field
- the draft table uses `Use` and `Trigger` checkbox columns so the trigger topic is chosen inline instead of through a dropdown, and checking `Trigger` promotes that row into the selected-use set if needed
- after drafting completes, the page shows a compact green checkmark next to the `Draft spec` button instead of a larger draft summary card
- the preview step keeps the card within the viewport and scrolls the timestamp rows, field data, and presence blocks inside the preview area
- saving a new config returns the user to `/convert/use` with that config selected so it can be run immediately
- `/convert/use` is the saved-config flow with a top-level dropdown, selected-config details, and a `Run` action that uses the selected config
- selected assets are passed in via the `asset_ids` query param
- persist `saved_config_id`, `source_asset_id`, and `conversion_id` in the route so the page can deep-link and resume state
- after submit, the page shows conversion status, linked job status, output path, and reported output files, and the handoff confirmation is surfaced with a sonner toast instead of a status card
- active conversions are polled while the page stays open

Important limitation:

- the legacy execution-first surface still exists in `frontend/app/convert/conversion-page.tsx`, but the routed convert entry now points at the split create/use pages

## Data Layer And Shared Hooks

- `frontend/lib/api.ts`: backend request/response types plus fetch helpers
- `frontend/hooks/use-backend.ts`: SWR hooks for assets, dashboard, conversions, saved configs, jobs, outputs, tags, episodes, replay data, and cache revalidation helpers
- `frontend/lib/conversion-authoring.ts`: JSON parsing, spec summary helpers, and inspection summaries for the authoring UI
- `frontend/hooks/use-index-asset.ts`: indexing mutations and pending-state tracking
- `frontend/hooks/use-upload-assets.ts`: upload mutation flow
- `frontend/hooks/use-scan-directory.ts`: directory scan flow
- `frontend/hooks/use-create-conversion.ts`: conversion submit helper

## Dormant Or Partially Wired Code Paths

- `frontend/components/conversion-dialog.tsx`: older modal-style conversion UI that is not used by the active routed workflow
- `frontend/app/convert/conversion-page.tsx`: legacy execution-first conversion page, retained but no longer routed
- `frontend/components/rerun-viewer.tsx`: wrapper for the official Rerun web viewer, currently not used by the active replay route
- `frontend/hooks/use-backend.ts`: includes hooks for authoring capabilities, viewer-source, and prepare-visualization flows that are not currently consumed by the routed UI

## Current Limitations And Gotchas

- The conversion authoring workspace is functional, but there is still room for visual polish and future import/export affordances.
- Some backend concepts already exist in typed client code before they exist in the actual backend implementation. Example: `OutputActionType` includes `vlm_tagging`, but only `refresh_metadata` works today.
- Replay currently centers on timeline + sample data, not the backend viewer-source / cached `.rrd` flow.
- There is some duplication from earlier iterations, especially around conversion UI and viewer integration. The routed versions under `frontend/app/` are the active surfaces.

## Where To Start For Common Tasks

- Add or change a route: `frontend/app/`
- Change a backend contract or fetch helper: `frontend/lib/api.ts`
- Add or change a shared data hook: `frontend/hooks/use-backend.ts`
- Change shared layout or nav: `frontend/components/app-shell.tsx`
- Change shared UI primitives: `frontend/components/ui/`
